# Smart Flow Metering Engineering Guardrails

This file defines non-negotiable engineering standards for all agent and human code changes in this repository.

## 1) File Size and Decomposition

- Source files (`.ts`, `.tsx`, `.js`, `.jsx`) must stay at or below 250 LOC.
- If a file grows past 250 LOC, split by responsibility immediately.
- Prefer feature folders with focused modules:
  - `routes/<feature>/...`
  - `services/<feature>/...`
  - `lib/<domain>/...`
- Exceptions are generated files and third-party scaffolding.
- Explicit exclusions: `apps/*/src/components/ui/**` files generated by shadcn/ui can exceed 250 LOC unless actively customized for product behavior.

## 2) DRY by Default

- Do not duplicate business rules, validation logic, or response shaping.
- Shared logic must be extracted into helpers/services/middleware.
- Avoid copy-pasting endpoint handlers; compose shared functions instead.
- Keep one canonical implementation per concern (auth checks, timestamps, rate-limit keys, etc).

## 3) Security-First Rules

- Validate all external input at boundaries (query, params, body, headers, webhooks).
- Enforce auth and role checks explicitly on protected endpoints.
- Never trust client-provided IDs/status without server-side verification.
- Never log secrets, tokens, API keys, full auth headers, or raw credentials.
- Use least privilege for DB/data access; fetch only required columns.
- Fail closed on auth/security checks.
- Use rate limiting on public and payment-sensitive endpoints.
- Keep dependency risk low: prefer maintained packages and avoid unnecessary dependencies.

## 4) Performance and Reliability

- Prioritize low-latency I/O paths and short critical sections.
- Avoid repeated DB calls when one query can safely serve the same purpose.
- Keep webhook callbacks fast and offload heavy work to queues.
- Use idempotency keys or unique job IDs for externally-triggered async flows.
- Prefer bounded pagination and explicit limits.

## 5) Maintainability

- Use clear naming and small functions with single responsibility.
- Keep route files thin; move domain logic to services.
- Centralize cross-cutting concerns (error handling, auth, rate limit, logging).
- Keep comments only where intent is non-obvious; avoid noise comments.

## 6) Dependency and Version Policy

- Do not use unbounded dependency specifiers (`*`).
- Use semver ranges and align to current stable releases.
- Re-check latest versions before major migration work.
- Runtime target for backend is Node LTS unless explicitly changed.
- Docker Compose images must be pinned to explicit stable tags (avoid `latest`).
- Prefer immutable image digests for production deployments when operationally practical.
- Infrastructure services (Postgres, Redis, admin tooling) must not be publicly exposed by default; keep them internal and opt-in for local ops access only.

## 7) Testing Expectation

- Every behavioral change must include or update automated tests.
- High-risk flows (payments, token generation, auth, callbacks) require integration tests.
- Bug fixes must include regression tests.

## 8) Change Acceptance Checklist

Before finishing a change, verify:

- Files comply with 250 LOC rule (or are decomposed).
- No duplicated logic introduced.
- Input validation and auth are in place.
- Type-check/build passes.
- Security-sensitive logs are clean.
