# Smart Flow Metering Engineering Guardrails

This file defines non-negotiable engineering standards for all agent and human code changes in this repository.

## 1) File Size and Decomposition

- Source files (`.ts`, `.tsx`, `.js`, `.jsx`) must stay at or below 300 LOC.
- If a file grows past 300 LOC, split by responsibility immediately.
- Prefer feature folders with focused modules:
  - `routes/<feature>/...`
  - `services/<feature>/...`
  - `lib/<domain>/...`
- Exceptions are generated files and third-party scaffolding.
- Explicit exclusions: `apps/*/src/components/ui/**` files generated by shadcn/ui can exceed 300 LOC unless actively customized for product behavior.
- Files that do NOT need splitting (reviewed and accepted):
  - `apps/admin-dashboard/src/routes/demo/sentry.testing.tsx` — demo/test scaffolding, not product code.
  - `apps/admin-dashboard/src/routes/demo/table.tsx` — demo scaffolding, not product code.
  - `apps/api/tests/e2e/gomelong-token.e2e.test.ts` — e2e test file; splitting tests harms readability.
  - `apps/public-website/src/components/contact/ContactInfo.tsx` — borderline at ~307 LOC; split only if it grows further.
  - `apps/public-website/src/components/register/RegisterFormSections.tsx` — borderline at ~302 LOC; split only if it grows further.
  - Build output files under `dist/` are excluded by definition.

## 2) DRY by Default

- Do not duplicate business rules, validation logic, or response shaping.
- Shared logic must be extracted into helpers/services/middleware.
- Avoid copy-pasting endpoint handlers; compose shared functions instead.
- Keep one canonical implementation per concern (auth checks, timestamps, rate-limit keys, etc).

## 3) Security-First Rules

- Validate all external input at boundaries (query, params, body, headers, webhooks).
- Enforce auth and role checks explicitly on protected endpoints.
- Never trust client-provided IDs/status without server-side verification.
- Never log secrets, tokens, API keys, full auth headers, or raw credentials.
- Use least privilege for DB/data access; fetch only required columns.
- Fail closed on auth/security checks.
- Use rate limiting on public and payment-sensitive endpoints.
- Keep dependency risk low: prefer maintained packages and avoid unnecessary dependencies.

## 4) Performance and Reliability

- Prioritize low-latency I/O paths and short critical sections.
- Avoid repeated DB calls when one query can safely serve the same purpose.
- Keep webhook callbacks fast and offload heavy work to queues.
- Use idempotency keys or unique job IDs for externally-triggered async flows.
- Prefer bounded pagination and explicit limits.

## 5) Maintainability

- Use clear naming and small functions with single responsibility.
- Keep route files thin; move domain logic to services.
- Centralize cross-cutting concerns (error handling, auth, rate limit, logging).
- Keep comments only where intent is non-obvious; avoid noise comments.

## 6) Dependency and Version Policy

- Do not use unbounded dependency specifiers (`*`).
- Use semver ranges and align to current stable releases.
- Re-check latest versions before major migration work.
- Runtime target for backend is Node LTS unless explicitly changed.
- Docker Compose images must be pinned to explicit stable tags (avoid `latest`).
- Prefer immutable image digests for production deployments when operationally practical.
- Infrastructure services (Postgres, Redis, admin tooling) must not be publicly exposed by default; keep them internal and opt-in for local ops access only.

## 7) Testing Expectation

- Every behavioral change must include or update automated tests.
- High-risk flows (payments, token generation, auth, callbacks) require integration tests.
- Bug fixes must include regression tests.

## 8) Change Acceptance Checklist

Before finishing a change, verify:

- Files comply with 300 LOC rule (or are decomposed).
- No duplicated logic introduced.
- Input validation and auth are in place.
- Type-check/build passes.
- Security-sensitive logs are clean.

## 9) Domain Ownership Model

- The registered app-level user on the customer side is the landlord, not the tenant.
- A landlord owns their mother meter and is the only customer-side actor we explicitly model as an app user.
- Tenants do not own sub-meters in the system model.
- The system does not maintain tenant accounts as authenticated customer users.
- Tenant interaction is transactional only: we know tenants through token purchases and related payment/contact data, not through app ownership or account ownership of a sub-meter.
- Do not design customer-auth flows, device-token ownership, or notification ownership around tenants owning sub-meters.
- Customer-facing mobile/auth/device-token features must be modeled around the landlord and their mother-meter ownership unless the product requirements explicitly change.
- Tenant app access is still allowed, but tenant onboarding is not ownership-based.
- Tenant app onboarding should allow entry via sub-meter serial number so the tenant can access app functionality tied to that sub-meter.
- This tenant access model does not make the tenant the owner of the sub-meter in the core system model.
- Landlord onboarding is a separate flow and must be treated differently from tenant onboarding.
- When designing app auth, notification delivery, or device-token registration, distinguish clearly between:
  - landlord identity and mother-meter ownership
  - tenant app access via sub-meter serial onboarding

## 10) Docker Workflow Guardrails

- Do not run Docker commands when the user has said Docker is broken or storage-constrained until the user explicitly confirms it is fixed.
- Before rebuilding images, inspect likely impact first and avoid rebuilds unless code or image configuration actually changed.
- Prefer one clean rebuild followed by all relevant checks, instead of repeated incremental rebuilds.
- Do not copy test files or ad hoc assets into running containers as part of the normal workflow.
- Do not use `docker compose up --force-recreate` unless image/config changes require container recreation.
- Keep Docker build context small and respect `.dockerignore` to reduce cache invalidation and disk usage.
- Treat Docker build cache as a real storage cost; repeated multi-stage Node builds can consume large disk space quickly.
- When debugging Docker disk usage, prefer the official inspection/prune flow:
  - `docker system df`
  - `docker builder prune`
  - `docker image prune`
  - `docker system prune`
- Prefer official Docker guidance for cache optimization, build context control, and cleanup behavior:
  - https://docs.docker.com/build/cache/optimize/
  - https://docs.docker.com/reference/cli/docker/system/df/
  - https://docs.docker.com/reference/cli/docker/builder/prune/
  - https://docs.docker.com/reference/cli/docker/image/prune/
  - https://docs.docker.com/reference/cli/docker/system/prune/

## 11) Research-First Rule When Unsure

- If the correct implementation path is unclear or something is puzzling, look it up on the web before guessing.
- Prefer official documentation first, then high-quality primary sources for the specific tool, framework, or platform in use.
- Do not improvise infrastructure, Docker, platform, or library workflows when the expected behavior can be verified from documentation.
